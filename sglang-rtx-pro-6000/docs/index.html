SOTA Benchmarks on Google Cloud G4 — 8-GPU machine

G4 is the brand name of Google's GPU VMs containing NVIDIA RTX PRO 6000 GPUs. These VMs come in sub-shapes with configurations of 1, 2, 4, and 8 GPUs per node — details on the G4 hardware reference page. G4 machines do not have GPU-to-GPU networking via standard RoCE RDMA, which makes it challenging to run large open-source LLMs like Kimi, GLM, DeepSeek, and Qwen.

We showcase optimized serving recipes on GKE with 1 or 2 nodes, covering both original checkpoints and NVIDIA's NVFP4 checkpoints. The RTX PRO 6000 is a Blackwell SM120 chip, and many attention kernels are not yet developed for it — despite these challenges, we have managed to get very good performance. Most results here are throughput-oriented, and these recipes have been implemented by Google Cloud customers at production scale.

We complement the multi-node setup with the GKE Inference Gateway (llm-d), which provides KV-cache-aware multi-node inferencing, demonstrated in multi-region and multi-cluster setups.

We also showcase Kimi-K2.6 with a very long, multi-step agentic benchmark (up to 250 steps) that uses HiCache to offload GPU KV cache to the host, and we have experimented with L3 cache on Google Cloud Managed Lustre. For agentic workloads we found that lowering concurrency/batch size can give a more optimal setting than multi-node pipeline parallelism — a research field that needs further exploration. The Kimi-K2.6 setup has been tested with the Inference Gateway on both SGLang and vLLM, and we made many improvements to llm-d and SGLang in the process. There are perhaps hundreds of failed experiments behind these results — we want to thank RadixArk (SGLang contributors) for their overwhelming support in coming up with these benchmarks.

Hardware

GCP G4: NVIDIA RTX PRO 6000 Blackwell (SM120) GPUs on Google's Titanium offload architecture.

SM120 · CC 12.0 96 GB GDDR7 + ECC 1792 GB/s bandwidth PCIe 5.0 ×16 · no NVLink 300 W TDP

Node topology — g4-standard-384

Dual-socket AMD Turin. Each NUMA domain: half the vCPUs and memory, four GPUs on PCIe Gen 5, one 200 Gbps SmartNIC.

Logical topology — exact lane mapping varies by platform.

This layout is why recipes set SGLANG_SET_CPU_AFFINITY=1 (pin workers to the local socket), NCCL_SOCKET_IFNAME=eth0,eth1 (use both NICs), and NCCL_P2P_LEVEL=SYS (no NVLink — peer transfers cross the socket link).

RTX PRO 6000 (SM120)

CUDA / Tensor / RT cores24,064 / 752 (5th gen) / 188 (4th gen)
Memory96 GB GDDR7 ECC · 1,792 GB/s · 384-bit
Peak FP4 (sparsity)~4,000 TOPS
Precision formatsFP4 · FP6 · FP8 · INT8 · FP16 · BF16 · TF32 · FP32

SM120 adds hardware NVFP4 (E2M1) with micro-tensor scaling — 16-element blocks with FP8 scale factors plus a global FP32 scale. This is what lets the modelopt_fp4 recipes fit trillion-parameter-class MoE models on one 8-GPU node.

Cluster setup

gkecluster/ on GitHub ↗

gkecluster/createCluster_template.sh provisions everything: 8896-MTU VPC, multi-networking, private GKE cluster, Blackwell GPU pool, NAT, Artifact Registry, and an isolated benchmark-client pool.

  1. Set the project (required).
    shell
    export PROJECT_NAME="your-gcp-project-id"
# optional overrides
export REGION="us-central1" ZONE="us-central1-a" CLUSTER_NAME="my-sglang-cluster"
export GCLOUD_PATH=$(which gcloud)   # template hardcodes the author's path
  2. Review CIDR ranges. Defaults: subnet 10.0.0.0/20, pods 10.1.0.0/20, services 10.1.16.0/20, master 172.16.0.16/28. Change them if they collide with your networks.
  3. Run it. Idempotent — existing resources are skipped.
    shell
    chmod +x createCluster_template.sh && ./createCluster_template.sh
  4. Add the Hugging Face secret most recipes expect.
    shell
    kubectl create secret generic hf-secret --from-literal=HF_TOKEN=your_token

    benchmarking_scripts/benchmark-dsv2.yaml (core)

...

    --backend sglang-oai \
    --base-url http://sglang-serving-on-master-opt:8000 \
    --model deepseek-ai/DeepSeek-V3.2 \

    shell

Troubleshooting

OOMLower --mem-fraction-static; confirm model + KV fits 96 GB × GPUs.
NCCL hangs (multi-node)VPC must be MTU 8896; check NCCL_SOCKET_IFNAME=eth0,eth1.
Noisy numbersVerify the client landed on the benchmark pool, not a GPU node.

Metrics

TTFT time to first token (prefill + queueing) · TPOT per-token decode pace after the first · ITL inter-token latency distribution · Accept length avg accepted draft tokens per speculative step. At 512-concurrency saturation, single-node median TTFT can legitimately reach minutes — it's queueing, not decode.

What makes it work

  • HiCache (ratio 2.0, write_through_selective) — multi-turn prefixes survive eviction → the 81% hit rate.
  • EAGLE3 — draft lightseekorg/kimi-k2.5-eagle3; dual-node swept to steps 4 / drafts 6.
  • LPM scheduling — batches by longest prefix match to maximize reuse.
  • SMG router — power_of_two across the two GCE replicas (to be replaced by the Inference Gateway).

GKE Inference Gateway

models/KimiK2.6/gke-inference-gateway/ on GitHub ↗

Prefix-cache-aware routing: requests sharing a prompt prefix land on the replica that already holds it in KV cache — maximizing reuse, minimizing TTFT. Shipped for both SGLang and vLLM backends.

Request flow across two G4 nodes (replica StatefulSet, 2 replicas, anti-affinity).

Setup

  1. Gateway API + networking.
    shell
    gcloud container clusters update <CLUSTER> --gateway-api=standard --zone=<ZONE>
# gke-l7-rilb needs a proxy-only subnet
gcloud compute networks subnets create <NAME>-proxy-only \
  --purpose=REGIONAL_MANAGED_PROXY --role=ACTIVE \
  --region=<REGION> --network=<VPC> --range=10.129.0.0/23
  2. GCS weights access. Enable Workload Identity + GCS Fuse CSI driver; grant roles/storage.objectViewer on the bucket to the sglang-sa Workload Identity and the default compute SA.
  3. Deploy replicas + EPP + routing.
    shell
    kubectl apply -f kimi-k2.6/sglang-kimik2.6-1node.yaml
kubectl apply -f kimi-k2.6/sglang-kimi-podmonitoring.yaml

helm install precise-prefix-cache-routing \
  oci://registry.k8s.io/gateway-api-inference-extension/charts/inferencepool \
  --version v1.5.0 -f inference-gateway/epp-values.yaml -n default

kubectl apply -f inference-gateway/gateway.yaml
kubectl apply -f inference-gateway/httproute.yaml
kubectl apply -f inference-gateway/inference-objective.yaml
kubectl get gateway kimi-inference-gateway   # wait for IP
  4. Client. Set VLLM_API_URL in vllm-chat-app.yaml to the Gateway IP and apply.

Network & NCCL

networkbench/ on GitHub ↗

G4 nodes carry two 200 Gbps Diorite SmartNICs. DRANET (8896-MTU VPC + Dynamic Resource Allocation) exposes both to pods — and the benchmarks confirm the aggregate bandwidth is real.

iperf3 190/190 Gbps duplexNCCL 37.9 GB/s ≈ 303 Gbps

DRANET setup

01_setup_network.sh · 02_create_cluster.sh · 03_deploy_dra.yaml (essence)
gcloud compute networks create dranet-vpc --subnet-mode=custom --mtu=8896
gcloud compute networks subnets create sub0 --network=dranet-vpc --range=10.10.0.0/16 --region=us-central1
gcloud compute networks subnets create sub1 --network=dranet-vpc --range=10.20.0.0/16 --region=us-central1
gcloud compute networks subnets create sub2 --network=dranet-vpc --range=10.30.0.0/16 --region=us-central1

gcloud container clusters create dranet-cluster --network=dranet-vpc --subnetwork=sub0 \
  --enable-ip-alias --enable-dataplane-v2 --zone=us-central1-b --num-nodes=1

gcloud container node-pools create g4-pool --cluster=dranet-cluster \
  --machine-type=g4-standard-384 --accelerator type=nvidia-rtx-pro-6000,count=8 \
  --enable-gvnic --num-nodes=2 --zone=us-central1-b \
  --node-labels=cloud.google.com/gke-networking-dra-driver=true \
  --additional-node-network network=dranet-vpc,subnetwork=sub1 \
  --additional-node-network network=dranet-vpc,subnetwork=sub2

# pods claim both NICs via DRA
apiVersion: resource.k8s.io/v1
kind: ResourceClaimTemplate
spec:
  spec:
    devices:
      requests:
      - name: netdev-request
        exactly: { deviceClassName: netdev.google.com, allocationMode: ExactCount, count: 2 }
run
kubectl apply -f 03_deploy_dra.yaml
kubectl apply -f 04_iperf_dranet.yaml && ./run_iperf_10x.sh          # iperf3 ×10
kubectl apply -f chiv01-nccl-bench.yaml && ./run_bench_10x.sh        # NCCL dual-NIC ×10
